# ADR-0005: JWT Access + 불투명 Refresh 토큰 회전 인증

## 제목

무상태 JWT Access Token + DB 세션 기반 불투명 Refresh Token 회전(rotation) 인증 채택

## 상태

승인됨 (Accepted) — 2026-07-15

## 배경

Phase 1은 회원가입/로그인/로그아웃/비밀번호 변경/토큰 갱신을 도입한다. Family Memory AI는
가족의 개인 기록을 다루므로 인증 자격(credential)의 탈취·재사용 위험을 구조적으로 낮춰야
한다(PRD §29). 동시에 대부분의 API 요청은 인증 검사만으로 통과해야 하므로, 매 요청마다
DB를 조회하는 무거운 세션 검증은 지연·부하 측면에서 부담이다.

요구되는 성질은 서로 상충한다:

- **빠른 인가**: 요청당 DB 왕복 없이 토큰만으로 사용자 식별.
- **즉시 무효화**: 로그아웃/비밀번호 변경/탈취 감지 시 세션을 확실히 끊을 수 있어야 함.
- **탈취 저항**: 브라우저 저장소 노출(XSS)로 장기 자격이 유출되지 않아야 함.
- **재사용 감지**: 유출된 갱신 토큰이 재사용되면 이를 탐지해 계정 전체를 방어.

단일 토큰 방식으로는 이 네 가지를 동시에 만족하기 어렵다. 짧게 살면 UX가 나쁘고, 길게 살면
무효화가 불가능하다.

## 결정

- **이중 토큰**을 사용한다.
  - **Access Token = 무상태 JWT**(`@nestjs/jwt`). 페이로드 `{ sub: userId, email }`, 만료
    `ACCESS_TTL`(기본 900초). `Authorization: Bearer <jwt>` 헤더로 전송한다. 전역
    `AccessTokenGuard`(`APP_GUARD`)가 모든 라우트를 기본 보호하고, `@Public()`이 붙은
    라우트(register/login/refresh)만 통과시킨다. 검증은 서명/만료만 확인하므로 DB 왕복이 없다.
  - **Refresh Token = 불투명 랜덤 값**(`randomBytes(32).toString('hex')`). DB엔 원문이 아니라
    `sha256(raw)` **해시만** `userSessions`에 저장한다. 전송은 **HttpOnly 쿠키**
    (`refresh_token`, `Path=/v1/auth`, `SameSite=Lax`, `Secure`는 production에서만,
    `Max-Age=REFRESH_TTL` 기본 30일)로만 이뤄진다 — JS로 읽을 수 없어 XSS로 유출되지 않으며,
    경로 스코프를 `/v1/auth`로 좁혀 다른 엔드포인트로 새지 않는다.
- **회전(rotation)**: `POST /v1/auth/refresh` 시 매칭된 기존 세션을 `revokedAt`으로 폐기하고,
  새 refresh 세션 + 새 access token을 발급한다. 갱신 토큰은 1회용이다.
- **재사용 탐지**: 제시된 raw refresh가 **이미 revoked된 세션**과 매칭되면 유출·재사용으로 간주해
  해당 사용자의 **모든 세션을 revoke**한 뒤 `401`을 반환한다.
- **즉시 무효화**: 로그아웃은 매칭 세션을 revoke(멱등). 비밀번호 변경은 **모든 세션을 revoke**해
  재로그인을 강제한다.
- **비밀번호 해시 = argon2id**(`@node-rs/argon2`, 프리빌트 바이너리, node-gyp 불필요). 기본
  파라미터를 사용한다.
- **정보 비노출**: 로그인 실패·미존재 계정·비활성 계정은 모두 동일한 일반 메시지의 `401`로
  응답해 자격 존재 여부를 노출하지 않는다. 로그·에러 메시지에 비밀번호/토큰/해시/연결문자열을
  절대 출력하지 않는다.

## 검토한 대안

1. **JWT 단일 토큰(장수명 access, refresh 없음)**: 요청당 DB 없이 간단하나, 로그아웃/비밀번호
   변경/탈취 시 **즉시 무효화가 불가능**하다. 만료까지 유효한 토큰이 남는다.
2. **JWT를 refresh로도 사용(장수명 JWT refresh)**: 서명만으로 검증되지만 서버가 보유 목록을
   가지지 않아 회전·재사용 탐지·강제 폐기를 구현할 수 없다.
3. **불투명 세션 토큰 단일 사용(access도 매 요청 DB 조회)**: 즉시 무효화는 쉬우나 **모든 요청이
   DB 왕복**을 유발해 지연·부하가 커진다. 무상태 access의 이점을 잃는다.
4. **Access Token을 localStorage에 저장**: 구현은 쉬우나 XSS 한 번으로 장기 자격이 탈취된다.
   본 결정은 refresh를 HttpOnly 쿠키에 두고 access는 메모리(짧은 수명)로 제한한다.
5. **비밀번호 해시로 bcrypt/scrypt/PBKDF2**: 널리 쓰이나 argon2id가 메모리-하드로 GPU/ASIC
   공격 저항이 우수하다. `@node-rs/argon2`는 프리빌트라 빌드 체인 부담도 없다.

## 장점

- Access는 무상태라 인가 경로에 DB 왕복이 없어 지연·부하가 낮다.
- Refresh는 DB 세션에 묶여 로그아웃/비밀번호 변경/탈취 감지 시 **즉시·확실한 폐기**가 가능하다.
- 원문 대신 해시만 저장하므로 DB 유출 시에도 갱신 토큰 원문이 노출되지 않는다.
- HttpOnly + 경로 스코프 쿠키로 XSS·경로 누수 표면을 줄인다.
- 회전 + 재사용 탐지로 갱신 토큰 도난 시 자동으로 계정 전체를 방어한다.
- 401 일반화로 계정 열거(enumeration) 공격을 차단한다.

## 단점

- 무상태 access는 만료 이전에는 개별 폐기가 불가능하다(무효화의 실질 지연 = `ACCESS_TTL`).
  → `ACCESS_TTL`을 짧게(기본 900초) 유지해 완화한다.
- `userSessions` 테이블·회전 로직이라는 상태 관리 복잡도가 추가된다.
- 회전은 동시 요청(여러 탭)에서 정상 갱신이 재사용 탐지로 오인될 여지가 있다(경합).
  Phase 1 범위에선 단순 회전으로 두고 필요 시 grace window를 검토한다.
- Refresh 쿠키의 `Path=/v1/auth` 스코프 때문에 클라이언트는 갱신을 해당 경로로만 호출해야 한다.

## 변경조건

- 다중 탭/오프라인 재접속에서 회전 경합이 실사용 문제가 되면 짧은 **grace window**(직전
  토큰 일정 시간 허용) 또는 refresh 토큰 패밀리(family) 추적을 도입한다.
- 세션 수가 많아져 `userSessions` 조회가 병목이 되면 인덱스 재점검 또는 만료 세션 정리
  (주기적 GC) 잡을 추가한다.
- 소셜 로그인/OIDC/2FA가 요구되면 토큰 발급 지점(`TokenService`)을 확장하되 회전·재사용
  탐지 원칙은 유지한다.
- 서비스가 HTTPS 종단 외부(모바일 네이티브 등)로 확장되면 쿠키 기반 refresh 대신
  보안 저장소 + 회전 방식으로 전송 계층을 재평가한다.
