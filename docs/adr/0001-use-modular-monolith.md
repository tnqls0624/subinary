# ADR-0001: 모듈러 모놀리스 아키텍처 채택

## 제목

모듈러 모놀리스(Modular Monolith) 아키텍처 채택

## 상태

승인됨 (Accepted) — 2026-07-15

## 배경

Family Memory AI는 가족 단위 사용자를 대상으로 하는 서비스로, 초기 트래픽 규모가 작고
개발 인력이 소수다. 반면 기능 도메인(기억/기록, 가계부, 일정, AI 검색)은 명확히 구분되며
Phase가 진행될수록 늘어난다. 운영 목표는 `docker compose up --build` 한 번으로 전체 스택이
안정적으로 기동하는 것이며, 개인정보를 다루므로 서비스 간 데이터 이동 경로를 최소화해야 한다.
초기부터 마이크로서비스로 분리하면 네트워크 경계·배포 파이프라인·관측 인프라 비용이
기능 개발 속도를 압도할 위험이 있다.

## 결정

- **모듈러 모놀리스**로 시작한다. 코드 경계는 pnpm workspace 패키지(`@family/*`)와
  NestJS 모듈(Database/Storage/Queue/Ai/Health/Dev)로 강제한다.
- 배포 단위는 실행 특성에 따라 최소한으로만 나눈다:
  - `apps/api` — HTTP 요청 처리 (NestJS 11 + Fastify 5, port 3001)
  - `apps/worker` — BullMQ 큐 소비 (port 3002, health 전용 HTTP)
  - `apps/web` — Next.js 프런트엔드 (port 3000)
- api와 worker는 코드를 `packages/*`로 공유하되 프로세스는 분리해, HTTP 지연과
  백그라운드 작업 부하를 상호 격리한다.
- 서비스 간 통신은 공유 DB + Redis 큐(BullMQ)로 한정하고, 내부 RPC를 도입하지 않는다.

## 검토한 대안

1. **마이크로서비스**: 도메인별 독립 서비스 + API 게이트웨이. 확장성은 높으나
   초기 팀 규모 대비 배포/관측/계약 관리 비용이 과도하다.
2. **단일 프로세스 모놀리스 (worker 통합)**: api 프로세스 내에서 큐 소비까지 수행.
   구성은 단순하지만 무거운 백그라운드 작업(향후 임베딩/파싱)이 HTTP 지연에 직접 영향을 준다.
3. **서버리스(FaaS)**: 저트래픽에 비용 효율적이나, 장시간 작업·로컬 개발 재현성·
   self-hosted(개인정보 보관) 요구와 맞지 않는다.

## 장점

- 로컬 1커맨드 기동, 단순한 배포·디버깅, 낮은 인프라 비용.
- 패키지/모듈 경계가 명확해 이후 서비스 분리(strangler 방식) 시 이동 단위가 이미 정의됨.
- api/worker 프로세스 분리로 부하 격리는 확보하면서 코드 중복은 없음.
- 트랜잭션이 단일 DB 안에서 완결되어 분산 트랜잭션 문제가 없음.

## 단점

- 단일 DB/코드베이스에 대한 결합이 커질 수 있어 경계 규율(모듈 간 직접 import 금지 등)을
  리뷰로 유지해야 한다.
- 특정 도메인만 독립적으로 수평 확장하기 어렵다(프로세스 단위 확장만 가능).
- 전체 재배포가 기본 단위가 된다.

## 변경조건

다음 중 하나가 현실화되면 해당 도메인부터 서비스 분리를 재검토한다:

- 특정 도메인(예: AI 검색/임베딩 파이프라인)의 부하가 나머지와 10배 이상 비대칭.
- 팀이 도메인별 독립 배포 주기를 요구할 만큼 성장.
- 단일 DB가 확장 한계에 도달(커넥션/스토리지/쿼리 경합)하여 도메인별 데이터 분리가 필요.
- 규제/보안 요구로 특정 데이터(예: confidential 등급)의 물리적 격리가 필요.
