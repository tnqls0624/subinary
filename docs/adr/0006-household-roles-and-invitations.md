# ADR-0006: 가족 그룹 역할 모델과 토큰 기반 초대

## 제목

가족 그룹(household) 4단계 역할 + 서비스 계층 인가 강제 + 해시 저장 1회용 초대 토큰 채택

## 상태

승인됨 (Accepted) — 2026-07-15

## 배경

Phase 1은 가족 그룹 생성/초대/역할 관리/초대 수락/구성원 제거를 도입한다(PRD §31). 핵심
불변식은 **가족 경계**다: 한 가족의 데이터는 그 가족의 구성원만 접근해야 하고, 다른 가족
구성원이나 비멤버는 차단되어야 한다(완료 조건 5, PRD §29). 또한 가족 내부에서도 초대·역할
변경·구성원 제거 같은 관리 작업은 아무나 할 수 없어야 한다(PRD §7.2).

초대는 특히 민감하다. 초대 링크(토큰)가 유출되면 외부인이 가족 데이터에 접근할 수 있으므로,
토큰은 짧은 수명·1회성·폐기 가능해야 하고, 유출 시에도 DB 원문이 그대로 노출되지 않아야 한다.
가족 합류는 개인정보 공유를 수반하므로 합류 시 **동의 기록**이 필요하다(PRD §7.3).

인가를 어디서 강제하느냐도 설계 포인트다. 컨트롤러/가드에서만 검사하면 새 진입점이 생길 때마다
검사가 누락될 위험이 있고, 서비스가 재사용될 때 경계가 무너진다.

## 결정

- **역할 모델**: `householdRole` = `owner | admin | member | viewer`(pgEnum). 역할은
  **가족 단위**로 부여된다(전역 역할 아님) — 한 사용자가 가족 A의 owner이면서 가족 B의
  viewer일 수 있다. `householdMembers`에 `(householdId, userId)` UNIQUE로 1인 1역할을 보장한다.
- **인가는 서비스 계층에서 `actorUserId` 기준으로 강제한다**(PRD §26). 모든
  `HouseholdService` 메서드는 호출자 `actorUserId`를 받고, 공통 헬퍼 `requireMembership(
  householdId, userId, roles?)`가 **활성 멤버십**을 조회한다:
  - 멤버가 아니면 `ForbiddenException('not a household member')`.
  - `roles`가 주어졌는데 역할이 불충분하면 `ForbiddenException('insufficient role')`.
  이로써 컨트롤러/가드를 우회하는 어떤 경로로도 가족 경계가 뚫리지 않는다(방어 심층화).
- **Phase 1 권한 매핑**(PRD §7.2):
  - `owner`: 전체 + 초대 생성/취소, 역할 변경, 구성원 제거, 설정 변경.
  - `admin`: 조회 + 멤버 목록/초대 목록 조회, 가족 이름 수정(owner|admin).
  - `member` / `viewer`: 조회.
  - write 계열(초대/역할/제거)은 **owner 중심**, 이름 수정은 owner|admin.
- **소유권은 이전하지 않는다(Phase 1)**. 초대·역할 변경 스키마에서 `owner`를 제외
  (`invitableRole = admin | member | viewer`)하고, 대상이 owner인 역할 변경은 거부한다.
  **마지막 owner 제거는 금지**한다(`BadRequestException`) — 가족이 고아가 되지 않도록.
- **초대 토큰**: 불투명 랜덤 raw 토큰을 발급하고 응답에 **1회만** 노출한다. DB엔 원문이 아니라
  `sha256(raw)` **해시만**(`tokenHash`, UNIQUE) 저장한다. 초대는 상태 기계로 관리한다:
  `pending → accepted | revoked | expired`.
  - 수락은 `POST /v1/household-invitations/:token/accept`, body `consent: true` **필수**
    (동의 없으면 `BadRequestException`).
  - `status !== 'pending'`(이미 수락/취소/만료) → 재사용 차단(`Gone`/`Conflict` 등 4xx).
  - `expiresAt < now` → `expired`로 마킹 후 4xx. 지정 이메일이 있으면 수락자 이메일과
    불일치 시 `ForbiddenException`.
  - 이미 멤버인 사용자가 수락하면 초대만 `accepted` 처리 후 기존 멤버십을 반환(멱등).
- **합류 동의 기록**: 가족 생성(owner)과 초대 수락 시 `householdConsents`에
  `consentType='household_join'`을 트랜잭션으로 함께 기록한다(PRD §7.3).
- **정보 비노출**: 비멤버 접근은 `403`(멤버십/권한). 로그·에러 메시지에 초대 토큰/해시를
  절대 출력하지 않는다.

## 검토한 대안

1. **전역 역할(사이트 단위 role)**: 단순하나 한 사용자가 여러 가족에 서로 다른 권한으로
   속하는 실제 요구를 표현하지 못한다.
2. **인가를 가드/데코레이터(`@Roles`)에서만 강제**: 선언적이라 읽기 좋으나, 서비스가 다른
   진입점(잡/내부 호출)에서 재사용될 때 검사가 누락된다. 본 결정은 **서비스 계층 강제**를
   1차로 두어 경계를 코드가 아니라 도메인 로직에 못박는다.
3. **정책 엔진(CASL/OPA 등) 도입**: 표현력은 높으나 Phase 1의 단순한 4역할 매트릭스에는
   과하고 의존성·러닝커브 비용이 크다. 규칙이 복잡해지는 후속 Phase에서 재검토한다.
4. **초대 토큰 원문을 DB에 저장**: 조회는 간단하나 DB 유출 시 유효 초대가 그대로 노출된다.
   해시 저장 + 1회 노출로 이 위험을 제거한다.
5. **이메일 기반 자동 합류(토큰 없이 이메일만으로 가입 시 자동 편입)**: 편하지만 오·남용
   위험이 크고 명시적 동의·만료·취소를 표현하기 어렵다.
6. **비멤버 접근에 404 반환(리소스 존재 은폐)**: 열거 방지엔 유리하나, Phase 1은 인증된
   사용자의 권한 실패를 `403`으로 명확히 구분한다(자격/멤버십 경계 표현). 초대 토큰 자체는
   해시·1회성이라 열거 위험이 낮다.

## 장점

- 가족 단위 역할로 다중 가족 소속을 자연스럽게 표현한다.
- 서비스 계층 인가로 **어떤 진입점에서도** 가족 경계·역할 규칙이 강제된다(누락 방지).
- 해시 저장 + 1회 노출 + 상태 기계로 초대 토큰의 유출·재사용·취소 위험을 구조적으로 낮춘다.
- 동의 기록(`householdConsents`)으로 개인정보 공유에 대한 감사 추적을 남긴다.
- 마지막 owner 보호·소유권 이전 제외로 Phase 1 범위에서 가족이 관리 불능 상태가 되는 것을 막는다.

## 단점

- 서비스 계층 인가는 각 메서드가 `actorUserId`와 `requireMembership`를 반복 호출해야 해
  보일러플레이트가 있다(헬퍼로 완화).
- 소유권 이전 미지원으로, owner 계정이 사라지면 Phase 1에선 수동 개입이 필요하다.
- 역할이 하드코딩된 매트릭스라 세밀한 권한(리소스별 세분화)이 필요해지면 재설계가 필요하다.
- 초대 만료를 서버 시간에 의존하므로, 검증 e2e는 시간 조작 대신 revoke로 대체 검증한다.

## 변경조건

- 권한 규칙이 리소스별로 세분화(가계부/일정/규칙별 권한)되면 정책 엔진 또는 권한 매트릭스
  테이블화를 재검토한다(후속 Phase).
- 소유권 이전·공동 owner 요구가 생기면 owner 승격 흐름과 마지막 owner 규칙을 확장한다.
- 초대 채널이 이메일 발송/딥링크로 확장되면 토큰 수명·재발송·rate limit 정책을 추가한다.
- 가족 규모가 커져 멤버십 조회가 병목이 되면 `householdMembers` 인덱스/캐싱을 재점검한다.
- `workspaces`(개인/회사 이벤트, Phase 6) 도입 시 역할 모델과의 상호작용을 재평가한다.
