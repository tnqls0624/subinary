# ADR-0019: 가맹점 사람 라벨 검토의 권한·확정 경계

## 상태

승인됨 (Accepted) — 2026-07-19

## 배경

가맹점 카테고리 학습 제어 평면과 준비도 gate는 마련됐지만 운영 데이터의 사람 확정 라벨은 아직 학습
기준보다 적다. 기존 거래 수정 화면은 개별 거래의 카테고리와 가맹점 규칙을 함께 바꿀 수 있으나, 학습용
라벨 수집이라는 목적과 AI 추천의 확정 여부가 명확하게 드러나지 않았다.

가족 구성원의 거래에는 `private`, `household`, `summary_only` 공개범위가 있다. 일반 거래 목록은
`summary_only`의 금액을 집계하면서 가맹점명을 마스킹하지만, 가맹점별 검토 큐가 이 행을 집계하면 원문 또는
동일 가맹점 존재를 역으로 노출할 수 있다. 또한 모델 prediction을 사용자 조작 없이 사람 라벨로 승격하면
자기학습 오류가 누적된다.

## 결정

### 1. 별도 라벨 저장소 대신 기존 규칙과 append-only feedback을 사용한다

검토 확인은 대표 거래에 `PATCH /v1/transactions/:id`와 `{ categoryId, applyRule: true }`를 보낸다. 기존
트랜잭션 안에서 거래 카테고리, `merchant_category_rules(source='human_confirmed')`,
`feedback_events(source='human_confirmed')`를 함께 기록한다. 같은 추천 카테고리를 확인한 경우도 명시적
사용자 확인 이벤트를 남긴다.

### 2. 검토 후보는 가맹점별로 묶고 원문 데이터를 최소화한다

승인 거래이면서 집계 제외되지 않았고 정규화 가맹점명이 있는 행만 후보가 된다. 사람·시스템 규칙이 이미
있는 가맹점은 제외하고, 규칙이 없거나 `model_prediction`인 가맹점만 반환한다. 응답은 정규화 가맹점명,
검토 가능한 거래 건수, 최근 시각, AI 추천 카테고리와 대표 거래 ID만 포함한다. 금액, 메모, 카드 문자
원문은 반환하지 않는다.

### 3. 열람 가능 범위보다 좁은 수정 가능·원문 공개 범위를 적용한다

- 일반 구성원: 본인 거래만 후보 집계에 포함한다.
- owner/admin: 본인 거래와 `visibility='household'`인 거래만 포함한다.
- 타인의 `private`와 `summary_only` 거래는 역할과 무관하게 후보에서 제외한다.

owner/admin이 원래 다른 구성원 거래를 수정할 수 있더라도, 원문 공개범위까지 확대하지 않는다.

### 4. AI 추천과 사람 확정을 UI와 계약에서 분리한다

`source='model_prediction'`은 선택창의 기본 추천값으로만 사용한다. 사용자가 카테고리를 선택하고 확인 버튼을
누르기 전에는 규칙 source와 feedback source를 변경하지 않는다. Training Runner는 계속
`human_confirmed` 라벨과 준비도 gate만 신뢰한다.

## 결과

### 장점

- 새 중복 저장소 없이 기존 데이터셋 builder의 계보와 revoke 규칙을 그대로 재사용한다.
- 타 구성원의 비공개 가맹점과 `summary_only` 존재 여부가 검토 큐로 새지 않는다.
- AI 추천을 확인하지 않은 상태와 사람이 동의한 상태를 명확히 구분한다.
- 라벨 수집 화면이 학습 준비도 부족의 직접적인 운영 해소 경로가 된다.

### 단점과 후속 작업

- 동일 household라도 구성원 역할과 공개범위에 따라 후보 건수가 다를 수 있다.
- 대표 거래 한 건에 확정 계보가 연결되므로 가맹점 규칙이 가리키는 전체 거래 집합은 snapshot 생성 시 다시
  계산해야 한다.
- 현재 큐는 최신순 작은 batch와 `hasMore`만 제공한다. 라벨이 크게 늘면 cursor, 검토 이력 필터와 category
  분포 기반 sampling을 별도 결정으로 추가한다.
